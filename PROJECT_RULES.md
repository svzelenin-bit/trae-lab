# PROJECT_RULES.md

## Назначение файла

Этот файл содержит правила работы с ИИ внутри репозитория `trae-lab`.

Его задача — сделать работу предсказуемой, безопасной, понятной и удобной для новичка в вайб-кодинге.

## Общие правила

- Отвечай по-русски.
- Пиши просто, понятно и без лишней воды.
- Сначала анализируй задачу, потом предлагай план.
- Не начинай менять код сразу без объяснения.
- Если задача неясна, сначала задай уточняющий вопрос.
- Не усложняй решение без необходимости.
- Для одной задачи не предлагай слишком много вариантов сразу.

## Правила перед изменением кода

Перед любым изменением файлов всегда выполняй такой порядок:

1. Кратко объясни, что ты понял.
2. Предложи план из 3 шагов.
3. Перечисли, какие именно файлы ты хочешь изменить.
4. Дождись подтверждения.
5. Только после подтверждения вноси изменения.

## Правила изменения файлов

- Меняй только те файлы, которые относятся к задаче.
- Не переписывай весь проект, если нужно исправить только один блок.
- Не удаляй рабочий код без явной причины.
- Не ломай существующую структуру проекта.
- Не переименовывай файлы и папки без необходимости.
- Если можно сделать маленькое изменение вместо большого — выбирай маленькое изменение.
- Сохраняй читаемость и простоту кода.

## Правила для новых файлов

- Если нужен новый файл, сначала объясни, зачем он нужен.
- Новые файлы должны иметь понятные названия.
- Не создавай лишние файлы “на всякий случай”.
- Если структура проекта меняется, кратко покажи новую структуру.

## Стиль работы с кодом

- Делай решение максимально простым для новичка.
- Избегай лишних библиотек и зависимостей.
- По возможности используй чистый HTML, CSS и JavaScript, если этого достаточно.
- Не добавляй сложную архитектуру там, где можно обойтись простым решением.
- Пиши код так, чтобы его было легко читать и редактировать позже.

## Как объяснять изменения

После каждого изменения обязательно:

- Кратко объясни, что было сделано.
- Перечисли изменённые файлы.
- Объясни, что проверить руками.
- Если есть риск, предупреди об этом заранее.

## Формат ответов

Предпочтительный формат ответа:

### 1. Что я понял
Коротко опиши задачу.

### 2. План
Дай план из 3 шагов.

### 3. Какие файлы будут изменены
Перечисли файлы.

### 4. Что будет сделано
Кратко объясни результат.

### 5. Что проверить
Напиши, что нужно проверить после изменений.

## Чего делать нельзя

- Нельзя менять всё подряд без плана.
- Нельзя предлагать “магические” решения без объяснения.
- Нельзя скрытно менять логику проекта.
- Нельзя добавлять зависимости без необходимости.
- Нельзя выполнять крупный рефакторинг без отдельного согласования.
- Нельзя делать вид, что задача выполнена, если есть сомнения или ошибки.
- Нельзя игнорировать уже существующий код и структуру проекта.

## Приоритеты

Приоритеты в таком порядке:

1. Понятность
2. Безопасность изменений
3. Простота
4. Рабочий результат
5. Скорость

## Базовый рабочий шаблон

Если пользователь пишет короткую задачу, сначала работай так:

> Проанализируй текущую задачу.
> Объясни, что ты понял.
> Предложи план из 3 шагов.
> Перечисли файлы, которые будут изменены.
> Не меняй код без подтверждения.

## Главный принцип

Лучше маленький понятный шаг, чем большой хаотичный результат.
